### 📖 GLOSSAIRE — LES MOTS CLÉS

<div align="center">

| Terme | Définition simple |
|:--|:--|
| 📦 **Repository (dépôt)** | Le dossier du projet hébergé sur GitHub |
| 📋 **Clone** | Télécharger une copie du dépôt sur ton PC |
| 💾 **Commit** | Sauvegarder tes modifications localement avec un message |
| 🚀 **Push** | Envoyer tes commits locaux vers GitHub |
| 📡 **Fetch** | Vérifier si de nouveaux changements existent en ligne (sans les appliquer) |
| 🔄 **Pull** | Télécharger ET appliquer les derniers changements depuis GitHub |
| 🌿 **Branch (branche)** | Une version parallèle du projet pour travailler sans risque |
| 🔀 **Merge** | Fusionner une branche dans une autre |
| ↩️ **Revert** | Annuler un commit précédent sans effacer l'historique |
| 📜 **History (historique)** | La liste de tous les commits passés |
| ⚡ **Stash** | Mettre de côté des modifications non terminées temporairement |

</div>

### 💾 COMMIT — Sauvegarder tes Modifications

*Un commit = une "photo" de ton travail à un instant T, avec un message qui explique ce que tu as fait.*  
*Tes modifications sont sauvegardées **localement** sur ton PC — pas encore en ligne.*

<div align="center">

| Étape | Action |
|:--|:--|
| 1️⃣ | GitHub Desktop affiche les fichiers modifiés à gauche |
| 2️⃣ | Coche/décoche les fichiers à inclure dans le commit |
| 3️⃣ | Champ **Summary** (en bas) → écris un message clair : `Ajout textures uniforme` |
| 4️⃣ | Champ **Description** (optionnel) → détails supplémentaires |
| 5️⃣ | Clique sur **Commit to main** (ou le nom de ta branche) |

</div>

> 💡 **Bons exemples de messages de commit :**  
> `Ajout du mod véhicule T-72` · `Correction bug spawn joueurs` · `Mise à jour liste mods`

```
┌──────────────────────────────────────────────────────────────┐
│  ❌ Mauvais message : "modif" / "test" / "aaaaa"             │
│  ✅ Bon message    : "Ajout de la carte Everon modifiée"      │
└──────────────────────────────────────────────────────────────┘
```

### 🚀 PUSH — Envoyer tes Commits sur GitHub

*Push = envoyer les commits que tu as faits localement vers le dépôt en ligne sur GitHub.*  
*Après un push, tout le monde peut voir tes modifications.*

```
┌──────────────────────────────────────────────────────────────┐
│                       PUSH — ÉTAPES                          │
├──────────────────────────────────────────────────────────────┤
│  1. Fais d'abord un ou plusieurs Commits                     │
│  2. En haut de GitHub Desktop → bouton "Push origin"         │
│  3. Attends quelques secondes                                │
│  4. ✅ Tes modifications sont en ligne sur GitHub !           │
│                                                              │
│  💡 Si le bouton dit "Fetch origin" → clique dessus d'abord  │
│     pour récupérer les dernières mises à jour, PUIS Push.    │
│                                                              │
│  ⚠️ Si le push est refusé → fais d'abord un Pull (voir      │
│     ci-dessous), résous les conflits, puis re-Push.          │
└──────────────────────────────────────────────────────────────┘
```

---

### 📡 FETCH — Vérifier sans Télécharger

*Fetch = demander à GitHub "y a-t-il de nouveaux changements ?" sans les appliquer.*  
*C'est une opération 100% sûre — elle ne modifie rien sur ton PC.*

<div align="center">

| Ce que Fetch fait | Ce que Fetch ne fait PAS |
|:--|:--|
| ✅ Vérifie si de nouveaux commits existent en ligne | ❌ N'applique pas les changements sur tes fichiers |
| ✅ Met à jour l'indicateur dans GitHub Desktop | ❌ Ne modifie rien dans ton dossier de travail |
| ✅ Te montre combien de commits tu as à récupérer | ❌ N'efface ni ne crée aucun fichier |

</div>

```
┌──────────────────────────────────────────────────────────────┐
│                      FETCH — ÉTAPES                          │
├──────────────────────────────────────────────────────────────┤
│  1. En haut de GitHub Desktop → "Fetch origin"               │
│  2. GitHub Desktop affiche "↓ X" si des commits sont dispo   │
│  3. Si des commits sont disponibles → fais un Pull           │
│     (voir section Pull ci-dessous)                           │
└──────────────────────────────────────────────────────────────┘
```

> 💡 **Bonne habitude :** Fais un **Fetch** à chaque fois que tu ouvres GitHub Desktop pour savoir si quelqu'un d'autre a modifié le projet.

---

### 🔄 PULL — Récupérer et Appliquer les Mises à Jour

*Pull = Fetch + application automatique des changements sur ton PC.*  
*Pull = "télécharge et intègre les derniers commits du dépôt en ligne".*

```
╔══════════════════════════════════════════════════════════════╗
║                      PULL — ÉTAPES                           ║
╠══════════════════════════════════════════════════════════════╣
║  1. En haut de GitHub Desktop → "Pull origin"                ║
║     (ou le bouton "Fetch origin" devient "Pull origin"       ║
║      après un Fetch qui a trouvé des changements)            ║
║                                                              ║
║  2. GitHub Desktop télécharge les nouveaux commits           ║
║                                                              ║
║  3. Tes fichiers locaux sont automatiquement mis à jour      ║
║                                                              ║
║  ✅ Ton PC est maintenant synchronisé avec GitHub !          ║
╚══════════════════════════════════════════════════════════════╝
```

<div align="center">

| Situation | Que faire ? |
|:--|:--|
| Tu veux juste **vérifier** s'il y a des mises à jour | **Fetch** |
| Tu veux **récupérer** les mises à jour sur ton PC | **Pull** |
| Tu as des modifications **non commitées** avant un Pull | Commit d'abord, puis Pull |

> ⚠️ **Toujours Pull avant de commencer à travailler sur les fichiers !**

</div>

---

### 🔀 FETCH vs PULL — La Différence en un Coup d'Œil

```
┌────────────────────────────────────────────────────────────────┐
│                   FETCH  vs  PULL                              │
├────────────────────────────┬───────────────────────────────────┤
│          FETCH             │             PULL                   │
├────────────────────────────┼───────────────────────────────────┤
│  Vérifie les nouveaux      │  Télécharge ET applique           │
│  commits en ligne          │  les nouveaux commits             │
│                            │                                    │
│  ✅ Sûr à 100%             │  ✅ Met à jour tes fichiers        │
│  ❌ Ne change rien         │  ⚠️ Peut créer des conflits        │
│                            │     si tu as des modifs locales    │
│  → Utilise pour vérifier   │  → Utilise pour synchroniser      │
└────────────────────────────┴───────────────────────────────────┘
```

---

### 🌿 BRANCH — Travailler en Parallèle sans Risque

*Une branche = une copie du projet sur laquelle tu travailles sans affecter la version principale.*  
*C'est comme travailler sur un brouillon avant de l'intégrer dans la version finale.*

```
╔══════════════════════════════════════════════════════════════╗
║                   POURQUOI UTILISER UNE BRANCHE ?            ║
╠══════════════════════════════════════════════════════════════╣
║  ✅ Tester une idée sans casser le projet principal          ║
║  ✅ Travailler sur plusieurs fonctionnalités en même temps   ║
║  ✅ Proposer des changements avant qu'ils soient validés     ║
║  ✅ Garder la branche "main" toujours stable                 ║
╚══════════════════════════════════════════════════════════════╝
```

<div align="center">

**Créer une nouvelle branche dans GitHub Desktop :**

| Étape | Action |
|:--|:--|
| 1️⃣ | En haut → clique sur le nom de la branche actuelle (ex : **main**) |
| 2️⃣ | Clique sur **New Branch** |
| 3️⃣ | Donne un nom à ta branche (ex : `mod-nouvelles-textures`) |
| 4️⃣ | Clique sur **Create Branch** |
| 5️⃣ | Tu travailles maintenant sur cette branche — **main** est intact ! |

**Changer de branche :**

| Étape | Action |
|:--|:--|
| 1️⃣ | En haut → clique sur le nom de la branche actuelle |
| 2️⃣ | Sélectionne la branche voulue dans la liste |

</div>

> 💡 **Conseil :** nomme tes branches de façon explicite : `mod-vehicules`, `fix-spawn-map`, `test-nouvelles-armes`

---

### 🔀 MERGE — Fusionner une Branche

*Merge = intégrer le travail d'une branche dans une autre (généralement dans **main**).*

```
┌──────────────────────────────────────────────────────────────┐
│                     MERGE — ÉTAPES                           │
├──────────────────────────────────────────────────────────────┤
│  1. Va sur la branche qui REÇOIT le merge (ex : main)        │
│     → En haut, clique sur la branche → sélectionne "main"   │
│                                                              │
│  2. Dans le menu → Branch → Merge into current branch…       │
│                                                              │
│  3. Sélectionne la branche à fusionner                       │
│     (ex : ta branche "mod-nouvelles-textures")               │
│                                                              │
│  4. Clique sur "Merge … into main"                           │
│                                                              │
│  5. ✅ Les modifications sont intégrées dans main !          │
│     N'oublie pas de Push pour envoyer sur GitHub.            │
└──────────────────────────────────────────────────────────────┘
```

> ⚠️ **Conflits de merge :** Si deux personnes ont modifié le même fichier, Git te demandera de choisir quelle version garder. GitHub Desktop t'indique les fichiers en conflit — ouvre-les et résous les conflits manuellement.

---

### ↩️ REVERT — Annuler un Commit

*Revert = annuler les effets d'un commit passé en créant un nouveau commit d'annulation.*  
*L'historique est préservé — rien n'est effacé définitivement.*

<div align="center">

| Étape | Action |
|:--|:--|
| 1️⃣ | Dans GitHub Desktop → onglet **History** (en haut à gauche) |
| 2️⃣ | Trouve le commit que tu veux annuler |
| 3️⃣ | Fais un **clic droit** sur ce commit |
| 4️⃣ | Sélectionne **Revert changes in commit** |
| 5️⃣ | Un nouveau commit d'annulation est créé automatiquement |
| 6️⃣ | Fais un **Push** pour envoyer l'annulation sur GitHub |

</div>

```
┌──────────────────────────────────────────────────────────────┐
│  REVERT vs RESET — Quelle différence ?                       │
├──────────────────────────────────────────────────────────────┤
│  ↩️ REVERT  → Annule un commit en créant un nouveau commit   │
│              ✅ Sûr, l'historique est conservé               │
│              ✅ Recommandé pour les projets partagés          │
│                                                              │
│  🗑️ RESET   → Efface des commits de l'historique            │
│              ⚠️ Dangereux sur les projets partagés !         │
│              ❌ À éviter sauf si tu sais exactement           │
│                 ce que tu fais                               │
└──────────────────────────────────────────────────────────────┘
```

---

### ⚡ STASH — Mettre de Côté des Modifications

*Stash = sauvegarder temporairement des modifications non commitées pour y revenir plus tard.*  
*Utile quand tu dois changer de branche sans vouloir commiter un travail inachevé.*

```
┌──────────────────────────────────────────────────────────────┐
│                     STASH — ÉTAPES                           │
├──────────────────────────────────────────────────────────────┤
│  1. Dans GitHub Desktop → menu "Repository"                  │
│     → "Stash all Changes"                                    │
│                                                              │
│  2. Tes modifications sont mises de côté                     │
│     (elles n'apparaissent plus dans les fichiers modifiés)   │
│                                                              │
│  3. Pour les récupérer :                                     │
│     → Menu "Repository" → "Pop Stash"                        │
│     → Tes modifications reviennent dans tes fichiers         │
└──────────────────────────────────────────────────────────────┘
```
